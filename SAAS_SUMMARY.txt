  ROMAA_BE → SaaS CONVERSION - QUICK SUMMARY
  Strategy: Database-per-Tenant
================================================================================

WHAT WE'RE DOING
----------------
Convert single-tenant Romaa_BE into multi-tenant SaaS where each customer
organization gets its own isolated MongoDB database.

  Master DB          → Organizations, Subscriptions, SuperAdmins, Plans
  Tenant DB (×N)     → All business data per org (Employees, Tenders, etc.)


================================================================================
  KEY POINTS
================================================================================

1. ISOLATION
   Each org = separate MongoDB database. Zero risk of data leakage.

2. CONNECTION MANAGER
   LRU-cached mongoose connections (max 100 active, auto-evict idle).

3. MODEL REFACTOR
   All ~60 models change from:
      mongoose.model("X", schema)
   to:
      getXModel(connection) → connection.model("X", schema)

4. AUTH FLOW
   JWT carries organizationId → middleware loads tenant DB → req.db attached.

5. ONBOARDING
   Signup creates org + provisions new DB + seeds defaults automatically.

6. CRONS
   Loop over all active tenants; run job per connection.

7. BILLING
   Stripe/Razorpay + subscription gating middleware.

8. FILE STORAGE
   S3 keys prefixed with org slug for logical isolation.

================================================================================
  TOP RISKS
================================================================================

  1. Connection explosion         → LRU pool + idle eviction
  2. Schema migrations per tenant → Automated migration runner
  3. No cross-tenant queries      → ETL to data warehouse
  4. Cron serial bottleneck       → BullMQ worker concurrency
  5. Accidental data leakage      → Integration tests with 2 test orgs
  6. Zero tests currently         → Jest foundation in Tier A
  7. Existing customers re-login  → Force token rotation on cutover
