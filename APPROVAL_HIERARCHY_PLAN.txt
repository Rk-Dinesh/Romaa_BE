0. EXECUTIVE SUMMARY
================================================================================

Romaa has a PARTIALLY-BUILT approval system. The finance module
(src/module/finance/approval/) already contains a working threshold-based,
multi-level approval workflow with audit logs, events, and email
notifications. BUT:

  - It is SCOPED to finance voucher source_types only.
  - It routes PURELY BY AMOUNT (threshold bands).
  - It does NOT understand organizational hierarchy (reportsTo, department,
    designation).
  - Other modules (HR-Leave, Tender, Purchase, WorkOrder, DLP, NMR, Weekly
    Billing, EMD) each re-implement ad-hoc approval patterns with
    inconsistent status enums and no shared audit trail.

GOAL:
Promote the finance approval engine into a GENERIC, REUSABLE
"ApprovalHierarchy" subsystem that ALL modules plug into, supporting:
  (a) amount-threshold routing (current finance behaviour),
  (b) role/designation-based chains,
  (c) manager-chain (reportsTo) dynamic resolution,
  (d) department/project-scoped approvers,
  (e) delegation (out-of-office),
  (f) SLA/escalation,
  (g) parallel ("any_of") + sequential + hybrid chains,
  (h) unified audit trail + notifications + events.

The approach is ADDITIVE — the existing finance/approval module gets
hoisted to a shared location and extended. No breaking changes on day 1.

2. GAPS & PROBLEMS WITH CURRENT APPROACH
================================================================================

G1. Approval engine scoped to finance only; non-finance modules cannot
    reuse it without depending on src/module/finance/.

G2. Only threshold(amount)-based routing. Cannot express:
      - "Manager of initiator must approve first"
      - "Project Manager of projectId X must approve"
      - "All approvers of department Y must approve"
      - Chains that depend on entity context, not amount.

G3. No delegation. If an approver is on leave, the request stalls.

G4. No escalation / SLA. A pending request can sit forever.

G5. Hardcoded 2-level status enum in LeaveRequest, EMD, etc. Each module
    invents its own field names, making reporting and dashboards brittle.

G6. approvalRequest (source_type, source_ref) is a compound UNIQUE index —
    so you can only ever have ONE approval request per document. If a
    document is rejected and needs resubmission, the old request must be
    withdrawn or replaced.

G7. Finance approval emails use nodemailer directly and skip
    NotificationModel (no in-app notifications, no audit of dispatch).

G8. AuditLogService lives under src/module/finance/audit but is generic —
    fine to hoist.

G9. ApprovalRule.thresholds supports amount bands but the approvers list
    contains STATIC Employee ObjectIds. This breaks the moment an
    employee leaves or is reassigned — rules must be manually re-edited.

G10. No idempotency on approve/reject. A double-click could cause double
     log entries (though status guard partially protects).

G11. No support for CONDITIONAL chains (e.g. "skip Level 2 if amount < 10k
     AND project is internal").

G12. No generic UI-friendly "My Approvals" that spans modules (only
     finance's pendingForUser exists).

3.2 CORE CONCEPTS
-----------------

  ApprovalHierarchy (policy)
    The definition of HOW a given source_type gets approved.
    A hierarchy is a list of STAGES. Each stage resolves at runtime to a
    set of approvers. Stages run sequentially; approvers within a stage
    can be parallel (any_of) or all-required.

  ApprovalStage (sub-doc)
    stage_no        : Number (1,2,3,...)
    label           : String ("Manager", "Finance Head", "CFO")
    approver_mode   : enum [
                        "USER",            // explicit user list
                        "ROLE",            // any user holding role X
                        "DESIGNATION",     // any user with designation X
                        "DEPARTMENT_HEAD", // head of dept of initiator
                        "MANAGER_CHAIN",   // initiator.reportsTo^N
                        "PROJECT_MANAGER", // PM of the linked project
                        "INITIATOR_MANAGER",// direct reportsTo
                        "AMOUNT_BAND"      // legacy — threshold match
                      ]
    approvers[]     : [Employee]           (when mode=USER)
    role_ids[]      : [Role]               (when mode=ROLE)
    designation     : String               (when mode=DESIGNATION)
    manager_level   : Number               (for MANAGER_CHAIN, e.g. 2 = grandboss)
    any_of          : Boolean              (within this stage)
    min_approvers   : Number               (e.g. "2 of 4 any_of")
    condition       : Mixed                (optional JSON rule — see 3.4)
    sla_hours       : Number               (stage-level SLA)
    escalate_to     : enum [same list as approver_mode]
    escalate_target : Mixed                (matches approver_mode of escalate_to)

  ApprovalHierarchy (top level)
    hierarchy_id    : String      (AH-001 via IdcodeServices)
    source_type     : String      (e.g. "LeaveRequest", "PurchaseBill",
                                   "Tender", "WorkOrderRequest", ...)
    label           : String
    scope           : {                     (OPTIONAL filtering)
                        project_ids[]       // only applies to these projects
                        department[]        // only to requests from these depts
                        designation[]       // only if initiator has these
                        amount_min / amount_max
                      }
    stages[]        : [ApprovalStage]
    priority        : Number      (lower = matched first; used when multiple
                                   hierarchies match)
    is_active       : Boolean
    created_by / updated_by
    Indexes: (source_type, is_active, priority)

  ApprovalRequest (runtime instance)
    Extends the existing finance model with:
      hierarchy_id      : ref ApprovalHierarchy
      hierarchy_snapshot: Mixed          (full stages captured at initiate)
      current_stage_no  : Number
      stage_state[]     : [{
                              stage_no, required_approvers[],
                              approved_by[], rejected_by, status,
                              started_at, completed_at
                          }]
      initiator_id      : Employee
      entity_context    : Mixed          ({ project_id, department,
                                            amount, custom })
      sla_due_at        : Date           (first stage SLA deadline)
      escalated         : Boolean
      correlation_id    : String

    Drop the (source_type, source_ref) UNIQUE index; replace with:
      (source_type, source_ref, status) — non-unique; optionally a PARTIAL
      unique where {status:"pending"} to prevent two open approvals.

  ApprovalDelegation
    from_user    : Employee
    to_user      : Employee
    valid_from   : Date
    valid_until  : Date
    source_types : [String]    (optional — scope to specific docs; empty = all)
    note         : String
    is_active    : Boolean
    Indexes: (from_user, is_active, valid_from, valid_until)

3.3 RESOLUTION ALGORITHM
------------------------
Given an entity to approve:

  1. Controller calls ApprovalService.initiate({source_type, source_ref,
     source_no, initiator_id, entity_context}).
  2. Service calls ApprovalResolverService.pickHierarchy(source_type,
     entity_context). This queries ApprovalHierarchy where
        is_active=true AND source_type=X
     filters by scope predicates against entity_context, sorts by priority,
     returns first match. If none, auto-approve or error (configurable).
  3. For each stage in hierarchy.stages:
        resolveStageApprovers(stage, entity_context, initiator) returns
        a de-duplicated Employee[] list, applying delegation.
  4. Snapshot the resolved stages into ApprovalRequest.stage_state[] and
     set current_stage_no=1.
  5. Persist. Emit approval.requested event. Create Notification
     (category=approval) targeted at stage 1 approvers. Log audit.

3.4 CONDITION LANGUAGE (OPTIONAL)
---------------------------------
Keep it simple — a JSON predicate like:
    { "amount": { ">=": 100000 } }
    { "project_type": { "in": ["Internal","R&D"] } }
    { "$and": [ {"amount":{">":50000}}, {"department":{"=":"Purchase"}} ] }
Evaluated with a small pure function; no eval(). Applied (a) at hierarchy
match time (scope) and (b) at stage level (skip stage if false).

3.5 STAGE TRANSITIONS
---------------------
  - approve():
      * actor must be in stage_state[current_stage_no].required_approvers
      * push to approved_by
      * if any_of OR approved_by.length >= min_approvers
          -> mark stage complete, advance current_stage_no++
      * if no more stages -> ApprovalRequest.status = "approved",
          emit approval.approved, update source document status via
          registered callback (see 3.6)
  - reject():
      * any approver in current stage can reject -> terminal
      * ApprovalRequest.status = "rejected"; emit approval.rejected
  - withdraw(): only initiator, only when status=pending
  - comment(): any participant; does not change state
  - escalate() (cron or manual): SLA breach -> send to stage.escalate_to

3.6 SOURCE-DOCUMENT CALLBACK REGISTRY
-------------------------------------
Each module registers a handler so the approval engine knows how to mutate
the underlying doc on approve/reject:

    // at boot (app.js or module index)
    ApprovalService.registerHandler("LeaveRequest", {
      onApproved : async (req, ctx) => { /* set leave status Approved */ },
      onRejected : async (req, ctx) => { /* set leave status Rejected */ },
      onWithdrawn: async (req, ctx) => { /* revert */ },
    });

This keeps approval engine decoupled from each domain model.

      action: "initiated"|"approved"|"rejected"|..., actor_id, meta})

      NotificationService.send({

        priority: amount > X ? "high" : "medium",
        module  : <module of source_type>,
        audienceType: "user",
        users  : stage.required_approvers,
        title  : `${source_type} ${source_no} needs your approval`,
        message: `Amount: ₹${amount}. Stage ${stage_no}: ${stage.label}`,

    fans out email.
  - Emit events via a renamed generic emitter:
        approval.requested / approved / rejected / escalated / withdrawn

Per-module wiring pattern (example — LeaveRequest):

  // in src/module/hr/leave/leave.service.js
  import ApprovalService from "../../approval/approval.service.js";

  class LeaveService {
    static async submit(data, user) {
      const leave = await LeaveModel.create({ ...data,
                      status: "Pending", initiator: user._id });
      await ApprovalService.initiate({
        source_type   : "LeaveRequest",
        source_ref    : leave._id,
        source_no     : leave.leave_code,
        amount        : data.days,        // "amount" used for SLA sizing
        initiator_id  : user._id,
        entity_context: {
          project_id : data.project_id,
          department : user.department,
          leave_type : data.leave_type
        }
      });
      return leave;
    }
  }

  // in app.js (once, at boot)
  ApprovalService.registerHandler("LeaveRequest", {
    onApproved : async (req) => {
        await LeaveModel.findByIdAndUpdate(req.source_ref,
               { status: "Approved",
                 finalApprovedBy: req.approval_log.at(-1).actor_id,
                 finalApprovalDate: new Date() });
    },
    onRejected : async (req) => {
        await LeaveModel.findByIdAndUpdate(req.source_ref,
               { status: "Rejected" });
    }
  });

9. NOTIFICATION & EVENT FLOW
================================================================================

  initiate() →
      AuditLog.log(action="initiated")
      emit("approval.requested", payload)
      Notification.send(approvers of stage 1)          // in_app + email

  approve() [not final] →
      AuditLog.log(action="approved", stage_no=N)
      emit("approval.stage.approved")
      Notification.send(approvers of stage N+1)

  approve() [final] →
      AuditLog.log(action="approved")
      emit("approval.approved")
      handler.onApproved(request, ctx)                 // updates source doc
      Notification.send(initiator)                     // congrats

  reject() →
      AuditLog.log(action="rejected")
      emit("approval.rejected")
      handler.onRejected(request, ctx)
      Notification.send(initiator)

  escalate() (from SLA cron) →
      AuditLog.log(action="escalated")
      emit("approval.escalated")
      Notification.send(stage.escalate_target approvers + original)

  withdraw() →
      AuditLog.log(action="withdrawn")
      emit("approval.withdrawn")
      handler.onWithdrawn(request, ctx)

Cron for SLA:
  utils/approvalSlaCron.js runs every 15 minutes; scans
      ApprovalRequest where status=pending, stage_state[current].sla_due_at
      <= now AND escalated=false → escalate(); mark escalated=true.
  Register in app.js (same pattern as dailyAbsenteeism).

11. EDGE CASES & SPECIAL RULES
================================================================================

  E1. Self-approval: If initiator appears as an approver, skip them
      automatically (configurable flag allow_self_approval, default false).

  E2. Duplicate approvers across stages: Should later stages still be
      required? Config flag collapse_duplicate_approvers (default true ->
      skip later stage if approver already approved earlier).

  E3. No active hierarchy matches source_type + scope:
      Config: default_policy ∈ {"auto_approve","auto_reject","error"}.
      Default "error" — forces admin to configure.

  E4. Rejected → resubmit: allow initiator to modify source doc and call
      initiate() again; the prior ApprovalRequest is status=rejected so
      the partialUnique index is free.

  E5. Delegation cycle: A→B→A. Detect and truncate at cycle; log warning.

  E6. Approver leaves company: Mark Employee.isDeleted=true (existing
      pattern); resolver must SKIP deleted employees; if a stage is left
      empty, auto-escalate.

  E7. Manager chain exceeds depth: cap manager_level resolution at 5 to
      avoid loops.

  E8. Concurrent approval race (two approvers in any_of click at the same
      time): wrap state transition in a findOneAndUpdate with status
      guard (status: "pending", current_stage_no: X) — MongoDB's
      single-document atomicity handles it.

  E9. Idempotency: dedupe via approval_log where action+actor_id+stage_no
      already exists within 5 seconds.

  E10. Withdrawing mid-stage: allowed only before any stage approves (or
       configurable).

  E11. Delegation + any_of stage: delegate of one approver is ADDED to
       the list but does not change min_approvers.

  E12. Amount currency: existing code uses Number rupees. Keep that; if
       multi-currency is added later, normalize before threshold compare.
