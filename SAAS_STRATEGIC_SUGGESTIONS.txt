PHILOSOPHY
----------
The biggest risk in this journey is NOT technical.
It is spending 6 months building multi-tenant infra, only to discover
customers wanted something else.

  RULE #1: Validate demand BEFORE migration.
  RULE #2: Ship to 3 paying single-tenant customers FIRST.
  RULE #3: Convert to SaaS only if demand is real.


================================================================================
  SECTION 1 - STRATEGIC / BUSINESS
================================================================================

[ ] Customer validation
      - Interview 10-15 construction company owners/MDs
      - Ask: "Would you pay Rs.X/month for this?" (not "would you use it?")
      - Confirm DB-per-tenant isolation is actually a selling point

[ ] Pricing anchor
      - Anchor on cost of 1 site engineer (Rs.40k-80k/month)
      - Makes Rs.15k/month SaaS feel cheap
      - Suggested tiers:
          Starter:     Rs. 4,999/mo   (5 users, 3 projects)
          Pro:         Rs.14,999/mo   (25 users, unlimited projects)
          Enterprise:  Rs.49,999/mo+  (unlimited + SSO + SLA)

[ ] Find 1-2 lighthouse / pilot customers
      - Offer 6 months free in exchange for:
          - Weekly feedback sessions
          - Case study / testimonial rights
          - Reference calls for future prospects

[ ] Don't build what you can buy
      - Payments: Stripe / Razorpay (not custom)
      - Email: SendGrid / Resend (not self-hosted SMTP)
      - Auth: Evaluate Clerk / Auth0 / Supabase Auth vs custom JWT
      - Search: Algolia / Meilisearch (when needed)
      - Analytics: Mixpanel / PostHog

[ ] Dogfood it
      - If Romaa is already your internal tool, keep using it
      - You are "Tenant Zero" - feel every bug before customers do


================================================================================
  SECTION 2 - TECHNICAL FOUNDATIONS (BEFORE MULTI-TENANT)
================================================================================

Priority: Do these BEFORE starting SaaS migration. Bad foundation + multi-
tenant = 10x worse bugs.

[ ] TypeScript migration
      - Single biggest long-term investment
      - Catches tenant-isolation bugs at compile time
      - Example: passing wrong req.db becomes a type error, not runtime bug
      - Effort: 3-4 weeks incremental migration

[ ] Input validation layer (Zod or Joi)
      - Already flagged in memory as missing P3 gap
      - Every controller validates req.body / req.params / req.query
      - Prevents garbage data from corrupting tenant DBs

[ ] Global error handler middleware
      - Already flagged in memory
      - Centralize all catch blocks into one Express error handler
      - Consistent error response shape across all 60 modules

[ ] Tenant-isolation integration tests FIRST
      - BEFORE writing any multi-tenant code, write tests that:
          1. Seed 2 orgs with different data
          2. Login as Org A, attempt to read Org B resources
          3. Must return 403/404/empty - NEVER leaked data
      - These tests will catch 90% of tenant isolation bugs

[ ] Repository pattern (optional but recommended)
      - Abstract DB access into repository classes
      - Swapping connection is ONE change, not 60 service files
      - Easier to mock in tests


================================================================================
  SECTION 3 - SECURITY HARDENING
================================================================================

[ ] Secret management
      - Move Stripe / AWS / JWT secrets OUT of .env
      - Use AWS Secrets Manager or HashiCorp Vault
      - Rotate secrets quarterly

[ ] Account lockout
      - Lock account after 5 failed logins (not just rate limit IP)
      - Current rate limit is IP-based; attacker with rotating IPs bypasses

[ ] Password policy
      - Minimum 12 chars, zxcvbn strength check
      - Password history (no reuse of last 5)
      - Force rotation every 90 days for admins

[ ] Dependency scanning
      - Enable Dependabot on GitHub (free)
      - Snyk / OWASP Dependency-Check weekly
      - Auto-PRs for vulnerable packages

[ ] Security testing
      - OWASP ZAP automated scans in CI
      - External penetration test before first enterprise customer
        (Budget: Rs.2-5 Lakh from firms like NotSoSecure, Payatu)

[ ] Per-tenant encryption at rest (Enterprise tier)
      - Envelope encryption with per-tenant KMS keys
      - AWS KMS or HashiCorp Vault Transit


================================================================================
  SECTION 4 - DEVOPS & INFRASTRUCTURE
================================================================================

[ ] Docker from day 1
      - Dockerfile + docker-compose.yml
      - Local dev matches production (no "works on my machine")

[ ] Infrastructure as Code
      - Terraform OR Pulumi
      - One command spins up staging, DR, new regions
      - Version controlled, peer reviewed

[ ] CI/CD pipeline
      - GitHub Actions:
          - Lint + typecheck on every push
          - Tests on every PR
          - Auto-deploy to staging on merge to main
          - Manual approval for production
      - Blue-green or canary deploys for zero downtime

[ ] Preview environments per PR
      - Reviewers test actual changes in a live environment
      - Tools: Vercel for FE, Fly.io / Railway for BE

[ ] Separate compute tiers
      - API servers  = fast-scale (K8s HPA or ECS)
      - Worker nodes = for crons, queues, long jobs
      - Don't run crons on API servers (breaks auto-scaling)

[ ] Observability stack
      - Sentry - error tracking (per-tenant tags)
      - DataDog / Grafana Cloud / New Relic - APM
      - Loki / CloudWatch - log aggregation
      - PagerDuty - incident alerting

[ ] MongoDB Atlas cost planning
      - Atlas is fastest to start but costs scale aggressively at 100+ DBs
      - Estimate: M10 cluster ~ $60/mo per region
      - Above 500 tenants: consider dedicated M30+ or self-hosted

[ ] Part-time DevOps contractor
      - Docker, K8s, Terraform are a distraction from product
      - Hire specialist (Rs.50k-1L/mo) for 3 months to set up foundation


================================================================================
  SECTION 5 - PRODUCT DIFFERENTIATORS (BEAT COMPETITORS)
================================================================================

[ ] Bilingual UI from day 1
      - Hindi + English minimum
      - Tamil / Telugu / Bengali for regional market
      - Competitors are English-only - big moat for regional players

[ ] Voice input on mobile
      - Site supervisor dictates work done in Hindi / regional language
      - Auto-fill forms via speech-to-text
      - Huge UX win for field staff

[ ] PDF intelligence
      - Upload govt tender PDF -> auto-extract BOQ, dates, EMD amount
      - Saves 2-3 hours per tender
      - Use GPT-4 Vision or Claude for parsing

[ ] Govt tender auto-import
      - Daily scrape: GeM, CPWD, PWD, state portals
      - Push matching tenders to subscribed orgs via email / WhatsApp
      - Subscription upsell: Rs.5k/mo for auto-import

[ ] Benchmarking (aggregate insights)
      - "Your material costs are 12% above industry average"
      - "Projects in Chennai complete 8% faster than Mumbai"
      - Anonymized aggregate data = premium feature
      - Only possible with multi-tenant critical mass

[ ] Mobile-first site app
      - Competition is likely laptop-only = big differentiator
      - Offline support mandatory (rural sites have poor signal)

[ ] Contractor / vendor network effect
      - Once 100+ contractors are on platform, orgs join just to access them
      - Ratings, reviews, verified profiles
      - This is your LONG-TERM moat

================================================================================
  SECTION 7 - METRICS TO TRACK FROM DAY 1
================================================================================

Product Metrics
[ ] Activation rate    - % of signups who create first tender in 7 days
[ ] Time to value      - signup -> first meaningful action (target <10 min)
[ ] Feature adoption   - which modules are unused? Kill or fix
[ ] DAU / MAU ratio    - sticky usage > 20%

Business Metrics
[ ] MRR / ARR          - monthly / annual recurring revenue
[ ] Gross churn        - % of customers who cancel per month
[ ] Net churn          - accounting for upsells (negative = healthy)
[ ] CAC                - customer acquisition cost
[ ] LTV / CAC ratio    - target > 3:1
[ ] NPS                - quarterly survey, target > 40

Support Metrics
[ ] Ticket volume per customer  - high = onboarding broken
[ ] First response time         - under 2 hours business hours
[ ] Resolution time             - under 24 hours for non-critical


================================================================================
  SECTION 8 - TRAPS TO AVOID
================================================================================

[ ] DO NOT over-build before Product-Market Fit
      - Skip Tier D/E add-ons until 10+ paying customers
      - Every feature = maintenance cost forever

[ ] DO NOT customize for one customer
      - They will ask. Politely say no.
      - Or charge 3-5x Enterprise pricing + dedicated team

[ ] DO NOT offer free tier without limits
      - Destroys margins
      - Time-limited trial (14-30 days) > free forever

[ ] DO NOT ignore retention
      - SaaS lives and dies on churn
      - 5% monthly churn = 46% annual churn = dead in 2 years
      - 1% monthly churn = healthy

[ ] DO NOT build your own auth from scratch for multi-tenant
      - Evaluate Clerk / Auth0 / Supabase Auth
      - Auth bugs in multi-tenant = security breach = company-ending

[ ] DO NOT build 100 shallow features
      - 10 deep, polished features > 100 half-baked ones
      - Construction users value reliability over novelty

[ ] DO NOT ignore onboarding
      - 80% of trial users quit in first session if confused
      - Invest in guided walkthroughs, sample data, tooltips

================================================================================
  FINAL HONEST ADVICE
================================================================================

The technical migration plan (SAAS_MIGRATION_PLAN.txt) is solid.
But technical excellence alone does not build a SaaS business.

What actually matters:

  1. Customer demand (not your assumption of demand)
  2. Distribution (how customers find you)
  3. Retention (keeping them)
  4. Pricing (capturing value created)
  5. Execution speed (shipping weekly, learning faster)

Technology is 20% of the work.
Product + GTM + Retention is 80%.

Build the minimum technical foundation. Get paying customers. Let THEIR
problems drive your roadmap - not this document.
